# ADR: YouTube Ad Detector MVP

## ADR-001: 모노레포 구조
결정: npm workspaces 기반 `apps/extension` + `apps/web` + `packages/shared`
이유: 확장·웹이 공용 타입과 룰 사전을 공유해야 하고, Next.js로 서버와 보고서 웹을 하나로 묶을 수 있으므로 apps 2개 + shared 1개가 최소 구성
트레이드오프: 초기 설정 비용. shared 비대화 위험 → 타입/룰/유틸 외 배치 금지

## ADR-002: 확장 프레임워크
결정: plasmo
이유: 사용자 지정. TS/React/HMR과 Manifest v3를 바로 지원
트레이드오프: 대안(vite-plugin-web-extension 등)보다 커뮤니티가 얇음

## ADR-003: 서버 스택
결정: Next.js 풀스택 (App Router, `/api` Route Handler)
이유: 보고서 SSR 페이지와 분석 API를 한 배포 단위로 묶을 수 있고 서버 로직이 단순
트레이드오프: 서버리스 환경에서 파일 쓰기 불가 → 보고서는 stateless로 설계

## ADR-004: 자막 수집 전략
결정: 1차 YouTube API(서버), 실패 시 2차 DOM 파싱(확장 content script) 폴백
이유: YouTube API가 안정적이지만 자막 미존재·권한 이슈가 있을 수 있음. 폴백으로 확장에서 DOM을 긁으면 사용자 세션 맥락 활용 가능
트레이드오프: 두 경로 유지 비용. 유튜브 DOM 변경 시 폴백 깨질 수 있음 → 모니터링 필요

## ADR-005: 보고서 저장 전략
결정: stateless — 요청마다 재분석, 서버 메모리 캐시만 사용 (DB/파일 저장 없음)
이유: MVP에서는 영구 저장 불필요. Vercel 등 서버리스 배포와 호환
트레이드오프: 동일 영상 재방문 시 재계산 비용. 룰/자막이 결정적이라 결과 재현 가능

## ADR-006: 보고서 URL
결정: `/report/{videoId}`
이유: videoId가 전역 유일 키. 캐시·공유 URL이 단순
트레이드오프: 룰 버전이 바뀌어도 URL 동일 → 룰 버전은 페이지 본문에 명시

## ADR-007: YOUTUBE_API_KEY 관리
결정: `apps/web/.env.local`에 위치, 서버 사이드에서만 읽음 (`NEXT_PUBLIC_` 접두사 금지)
이유: Next.js 기본 경로이고, 서버 전용 키가 apps/web 밖으로 새지 않음
트레이드오프: 모노레포 공용 env 사용 불가 → 다른 앱에서 재사용 시 복사 필요
작업: 현재 루트의 `.env.local`은 apps/web로 이동하고, 루트 `.env.local`은 삭제

## ADR-008: rule-based 탐지 방식
결정: **3-zone 가중치 스코어링** — rule 엔진이 자막을 스캔해 카테고리별 위험 가중치를 합산, 총점을 3개 구간으로 라우팅
 - **Low (정상)**: 하한 임계값 미만 → "정상 영상"으로 즉시 판정, 이후 단계 스킵
 - **Mid (애매)**: 중간 구간 → (MVP) "의심" 라벨로 보고서 노출 / (포스트-MVP) LLM에 판별 위임
 - **High (허위 의심)**: 상한 임계값 초과 → "허위 광고 의심"으로 즉시 판정, LLM 스킵
이유:
 - 단순 1차 필터 방식은 비매칭 영상 전체가 LLM 경로로 몰려 향후 비용 폭증
 - 단순 가중치(단일 임계값) 방식은 룰을 트리거하지 못한 미묘한 허위 광고가 "정상"으로 빠져 false negative 고착
 - 3-zone은 양 끝(확실한 정상/확실한 의심)에서 LLM을 스킵해 비용을 낮추고, 중간 구간만 LLM이 보완 → 비용과 재현율의 균형
트레이드오프:
 - 두 개의 임계값(low/high)을 실제 자막 샘플로 캘리브레이션해야 함 → 초기에는 보수적 기본값으로 시작, 실데이터 수집 후 조정
 - MVP 단계에서는 LLM이 없어 Mid 구간이 "의심" 라벨로만 표출됨 → UI 카피를 "추가 판별 필요" 뉘앙스로 구성
 - 룰 사전(키워드/정규식/컨텍스트 윈도우) 구체 설계는 여전히 구현 착수 시 실자막 샘플 기반으로 결정 — 본 ADR은 "스코어링 구조"만 확정, "룰 형태"는 ADR-009로 분리
후속:
 - ADR-009로 "룰 사전·매칭 방식" (키워드/정규식/컨텍스트 윈도우) 별도 결정 예정
 - `packages/shared`에 rule 엔진 + 임계값 상수 배치 (ADR-001 모노레포 구조 준수)
